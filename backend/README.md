# Yoshlar Backend (FastAPI)

Backend for the Yoshlar Agentligi monitoring CRM. See [docs/backend/modules.md](../docs/backend/modules.md) for the module map and [docs/roles/](../docs/roles/) for per-role API expectations.

## Stack

- FastAPI + Uvicorn
- PostgreSQL 16 (async via SQLAlchemy 2.x + asyncpg)
- Alembic migrations
- Pydantic v2 schemas + settings
- JWT auth (python-jose) + bcrypt (passlib)
- Redis (cache + rate limiting)
- pytest + pytest-asyncio

## Quick start

```bash
# 1. Install
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"

# 2. Configure
cp .env.example .env
# edit JWT_SECRET (>= 32 chars)

# 3. Start Postgres + Redis (from the repo root)
cd .. && docker compose up -d postgres redis && cd backend

# 4. Run migrations
alembic upgrade head

# 5. Seed the 5 demo accounts (idempotent)
python -m scripts.seed_users

# 6. Run the app
uvicorn app.main:app --reload --port 8000
```

## Database

PostgreSQL 16 (async via `asyncpg`). Local dev container in [`../docker-compose.yml`](../docker-compose.yml) exposes:

| Service | Port | Purpose |
|---------|------|---------|
| `postgres` | 5432 | dev database |
| `postgres_test` | 5433 | ephemeral test database (tmpfs) |
| `redis` | 6379 | cache + rate limiting |

Default `DATABASE_URL` already points at the dev container. Override in `.env` for hosted Postgres.

### Adding a new migration

```bash
# 1. Pull latest main FIRST — this keeps revision chains linear.
git pull origin master

# 2. Import the new SA model in alembic/env.py so Base.metadata sees it:
#    from app.modules.youth import models as _youth_models  # noqa: F401

# 3. Autogenerate and review the diff before committing.
alembic revision --autogenerate -m "add youth table"
alembic upgrade head
```

### Migration workflow rules

- **Always `git pull` before `alembic revision --autogenerate`** — the new migration's `down_revision` then points at the latest head, so no branching.
- **One migration per PR.** Multiple migrations make review harder and rebase nastier.
- **Review the autogenerated SQL.** Autogenerate misses things: CHECK constraints, index renames, `server_default`. Edit the migration file before committing.
- **Never edit an applied migration.** Once a migration is on `main` (and thus on someone's DB), it is immutable. Need to change something? Write a new migration.
- **Resolving a conflict (two devs created migrations from the same parent):** the second dev to merge rebases their branch, deletes their migration file, and reruns `alembic revision --autogenerate` after pulling. Or use `alembic merge -m "merge X + Y" <rev_a> <rev_b>` for true parallel branches.

## Layout

```
backend/
├── alembic/                # migrations
├── app/
│   ├── main.py             # FastAPI app + router registration
│   ├── core/               # config, security, deps, exceptions, logging, constants
│   ├── db/                 # base, session
│   ├── middleware/         # rbac, request_id, audit
│   └── modules/            # one folder per business module
│       └── <name>/{router,schemas,models,repository,service,tests}.py
├── tests/                  # unit + integration
├── pyproject.toml
└── .env.example
```

```bash
pytest --cov=app             # with coverage
```
